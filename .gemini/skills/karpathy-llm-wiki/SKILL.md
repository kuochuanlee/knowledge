# Karpathy LLM Wiki

使用 LLM 建立和維護個人知識庫。你管理兩個目錄：`raw/`（不可變更的原始資料）和 `wiki/`（編譯後的知識文章）。原始資料放進 raw/，你將它們編譯成 wiki 知識文章，而 wiki 會隨著時間累積。

Karpathy 的核心理念：
- 「由 LLM 撰寫和維護 wiki；由人類閱讀和提問。」
- 「wiki 是一個持續存在、不斷累積的產物。」

## 架構

分為三層，全部位於使用者的專案根目錄下：

**raw/** — 不可變更的原始資料。你只能讀取，絕對不能修改。依主題子目錄分類（例如：`raw/machine-learning/`）。

**wiki/** — 編譯後的知識文章。你擁有完全的所有權。依主題子目錄分類，只有一層：`wiki/<主題>/<文章>.md`。包含兩個特殊檔案：
- `wiki/index.md` — 全域索引。每篇文章佔一行，依主題分組，包含連結 + 摘要 + 更新日期。
- `wiki/log.md` — 只能附加的操作日誌。

**SKILL.md**（本檔案）— 結構定義層。定義結構和工作流程規則。

範本存放在相對於本檔案的 `references/` 目錄中。當你需要 raw 檔案、文章、封存頁面或索引的確切格式時，請讀取它們。

### 初始化

僅在第一次 Ingest 時觸發。檢查 `raw/` 和 `wiki/` 是否存在。只建立缺少的東西；絕對不要覆寫現有檔案：

- `raw/` 目錄（包含 `.gitkeep`）
- `wiki/` 目錄（包含 `.gitkeep`）
- `wiki/index.md` — 標題 `# Knowledge Base Index`，內文空白
- `wiki/log.md` — 標題 `# Wiki Log`，內文空白

如果 Query 或 Lint 找不到 wiki 結構，請告訴使用者：「請先執行一次 ingest 來初始化 wiki。」不要自動建立。

## Ingest (攝取)

讀取 `raw/` 中的原始資料，然後將其萃取並編譯成知識文章至 `wiki/` 知識庫中。永遠都要執行這兩個步驟，沒有例外。

### Fetch (資料歸檔與元資料建置)

**[嚴格限制]**：你被禁止使用任何網路爬蟲或工具自行抓取外部資料。知識的品質由人類全權控管。

1. **被動讀取**：當使用者要求 Ingest 時，使用者一定已經將高品質的來源檔案（如 .md, .pdf, .txt）放置在 `raw/` 目錄中，或者會在對話中直接貼上純文字給你。你的任務是「讀取與處理」這些已核准的內容，而非「上網搜尋」。

2. **主題分類**：檢查現有的 `raw/` 子目錄；如果主題夠接近，就直接放入該目錄。只有在領域截然不同時，才建立新的子目錄。

3. **檔案命名與元資料解耦 (Metadata Decoupling)**：
   - **保留原始檔名**：絕對禁止修改使用者提供的原始檔案名稱（例如使用者放入 `OpenAI-Report-2026.md`，就必須保持原名）。只有當使用者在對話中直接貼上「純文字」且未指定檔名時，你才需要根據內容生成一個具備描述性的檔名（如 `描述性代稱.md`）。
   - **強制注入 Frontmatter**：無論檔名為何，所有 `raw/` 內的 Markdown 檔案頂端，**必須**使用 YAML Frontmatter 格式記錄屬性（包含真實標題、收集日期、標籤等）。
   - **忠實保留內文**：正文部分必須保留原始文字。你可以清除格式噪音（如多餘空白、破損的 HTML 標籤），但絕對不可濃縮、總結或改寫原意。
   - 確切的 YAML 與排版格式，請嚴格遵守 `references/raw-template.md`。

4. **檔案格式特別守則：PDF 伴生代理模式 (PDF Sidecar Pattern)**：
   當處理 `.pdf` 檔案時，請嚴格執行以下實體與靈魂分離流程：
   - **保留實體**：絕對不可修改、移動或刪除原始的 `.pdf` 二進位檔案。
   - **建立代理檔**：在同目錄下，建立一個與 PDF 檔名完全相同，但副檔名為 `.md` 的代理檔案（例如：`attention.pdf` 的代理檔為 `attention.md`）。
   - **注入元資料與連結**：在該 `.md` 檔案頂端寫入 YAML Frontmatter，並在正文最開頭使用 Obsidian 嵌入語法 `![[原始PDF檔名.pdf]]` 建立文件連結。
   - **純文字萃取**：讀取 PDF 內容，將其純文字無損轉錄至 `.md` 檔案的正文中。在此階段僅做文字萃取，嚴禁對內容進行任何總結或歸納。

### Compile (知識萃取與編譯至 wiki/)

判斷新內容的歸屬與整合策略：

- **補充既有概念** → 若新資料與現有文章核心論點高度相關，請合併至該文章。更新受影響的段落，並將新來源加入 YAML 的 `raw_sources` 陣列中。
- **建立全新概念** → 若為新概念，在最相關的主題目錄中建立新文章。檔案命名必須使用精煉的「概念名稱」（如 `agentic-ai.md`），絕不可使用 raw 的檔案名稱。
- **跨領域與知識網** → 若概念跨越多個主題，放置在最核心的目錄中。強制要求在內文或 `See Also` 區塊，使用雙向連結語法 `[[相關文章檔名]]` 建立交叉參考。

**[事實衝突與除錯機制]**：
這些動作不是互斥的。若新原始資料與現有 wiki 內容相牴觸，**絕對不可直接覆蓋舊資料**。你必須保留雙方觀點，標註分歧，並精準註明各自的原始資料出處。若衝突存在於不同文章中，請在兩處都加上警語並交叉連結。

**[行內引用要求 (Inline Citation)]**：
在萃取與重組段落時，必須在陳述事實的句尾加上引用標籤，例如：`...這項技術提升了 20% 效能 ([[2026-04-29-tech-report]])`，確保每一個論點都能追溯回 `raw/` 區的單一真相來源。

文章確切格式請嚴格遵守 `references/article-template.md`。

### Cascade Updates (連鎖更新與知識圖譜同步)

在建立或更新主要文章之後，請執行受控的連鎖反應，但必須嚴格限制影響半徑 (Blast Radius)，以防改寫失控：

1. **追蹤一階關聯 (1st-Degree Graph Traversal)**：
   - 掃描與主文章處於同一個主題目錄下的相鄰文章。
   - 循著主文章內的雙向連結 (`[[ ]]`) 或 `See Also` 區塊，找出直接關聯的其他主題文章。
2. **精準更新 (Targeted Update)**：
   - 只有當新引入的原始資料（Raw）對這些關聯文章的「核心論點」產生實質影響（例如推翻舊觀念、提供新數據）時，才進行內容更新。
   - **更新要求**：除了修改內文並加上行內引用外，必須將該篇新來源的連結新增至目標文章 YAML 的 `raw_sources` 陣列中，並將 `date_updated` 修改為今日日期。
3. **邊界限制 (Boundary Constraints)**：
   - 不要進行二階或更深層的遞迴更新（不要去更新「被更新文章」的關聯文章），除非使用者明確指示。
   - 封存頁面（包含 `status: archive` 或位於特定 Archive 目錄下的檔案）永遠不可進行連鎖更新，它們是不可變動的歷史快照。

### Post-Ingest (攝取後：目錄維護與操作日誌)

完成知識萃取與連鎖更新後，必須執行以下收尾工作，以維持系統的健康與可追溯性：

1. **條件式更新 `wiki/index.md` (目錄降載原則)**：
   - **[嚴格限制]**：如果本次 Ingest 只是更新了現有的文章內容，絕對不可修改 `index.md`。
   - **新增觸發**：只有當本次操作建立了「全新的文章」或「全新的主題分類」時，才將新項目的雙向連結 `[[ ]]` 與一句話摘要，按格式新增至 `wiki/index.md` 中對應的區塊。

2. **附加時序紀錄至 `wiki/log.md` (圖譜化日誌)**：
   - 永遠以 Append（附加）模式，將本次操作紀錄寫入 `log.md` 的最底部。
   - 必須使用 Obsidian 的雙向連結語法 `[[ ]]` 來標示來源與受影響的檔案，確保日誌能融入知識圖譜。
   - 如果沒有發生連鎖更新，請省略 Cascaded 相關紀錄。
   - 確切格式請嚴格遵照 `references/log-template.md`。

## Query (查詢)

搜尋 wiki 知識庫並回答問題。觸發條件範例：
- 「關於 X 我知道些什麼？」
- 「總結與 Y 相關的所有內容」
- 「根據我的 wiki 比較 A 和 B」

### 步驟

1. 讀取 `wiki/index.md` 以鎖定相關的主題與文章，必要時可直接檢索特定主題目錄下的檔案。
2. 讀取這些文章的內容並綜合出具備邏輯性的答案。
3. **[嚴格限制]**：必須優先使用 wiki 內的知識，而非你自身的訓練資料。當你在答案中引用了某篇文章的觀點時，必須使用 Obsidian 雙向連結語法 `[[文章檔名]]` 標註來源。絕對不要使用傳統的相對路徑或標準 Markdown 連結。
4. 預設僅在對話視窗中輸出答案。除非使用者明確要求，否則不要寫入任何實體檔案。

### Archiving (查詢封存)

當使用者明確要求「封存 (Archive)」或「將答案儲存至 wiki」時：

1. **建立實體檔案**：將綜合出來的答案寫成新的 wiki 頁面。檔案名稱必須反映查詢主題（例如：`transformer-architectures-overview.md`），並放置在最相關的主題目錄下。
2. **遵守封存範本**：必須寫入 YAML Frontmatter，確切格式請嚴格遵守 `references/archive-template.md`。封存頁面不具備 `raw_sources` 屬性，而是具備 `wiki_sources`，裡面填入答案中所引用的 wiki 文章雙向連結。
3. **知識快照原則**：永遠建立全新頁面。絕對不可將封存內容合併到現有文章中，因為封存內容是「基於特定提問綜合出來的答案」，屬於特定時間點的知識快照。
4. **更新目錄**：將此新頁面新增至 `wiki/index.md`。在 Summary（摘要）欄位的前方必須明確加上 `[Archived]` 標籤。
5. **寫入日誌**：以 Append（附加）模式將本次封存動作記錄至 `wiki/log.md` 的最底部。確切格式請參照 `references/log-template.md` 中的 Query 區塊範例。

## Lint (健康檢查與維護)

對 wiki 進行品質檢查與知識重整。分為兩種擁有不同權限層級的類別：

### Deterministic Checks (決定性檢查，自動修復)

利用系統邏輯自動修復以下結構性問題：

**Index consistency (索引一致性)** — 比對 `wiki/index.md` 與實際的 `wiki/` 目錄檔案：
- 檔案存在但未列入索引 → 將其雙向連結 `[[檔名]]` 新增至最相關的主題區塊，並加上 `(no summary)` 佔位符。
- 索引條目 `[[檔名]]` 指向不存在的實體檔案 → 在索引中該行標記為 `[MISSING]`。切勿擅自刪除條目，交由使用者判斷是否為誤刪。

**Link & Metadata Integrity (連結與元資料完整性)** — 檢查所有實體檔案的內部結構：
- **雙向連結驗證**：掃描所有檔案正文與 YAML 陣列（如 `raw_sources`、`wiki_sources`）中的 `[[檔名]]` 語法。若目標檔案不存在，請在專案內搜尋相似檔名；若有唯一高度相似的檔案，自動修復連結名稱；若無或有多個，紀錄並回報給使用者。
- **YAML 格式驗證**：確保所有檔案（含 raw 代理檔與 wiki 實體檔）皆具備標準的 YAML Frontmatter，且必填欄位（如 title, tags, status）未遺失。若有遺失，根據檔案脈絡自動補齊。

**See Also (關聯網路)** — 針對知識圖譜連通性進行修補：
- 掃描並新增相關文章之間明顯缺失的 `[[ ]]` 交叉參考。
- 移除正文或 See Also 區塊中，指向確定已廢棄或徹底刪除檔案的死連結。

**Stub Creation (空節點建立)** — 針對知識網中的高頻盲區自動擴充：
- 掃描所有文章，若發現某個概念被頻繁提及或已有 `[[未建立概念]]` 的連結，卻尚未建立實體檔案，請自動建立一個基礎檔案。
- 該檔案只需包含 YAML Frontmatter（設定 `status: "stub"`），並在內文寫上「> 待編譯：此概念在知識庫中頻繁出現，尚未進行系統性統整。」，以待日後 Ingest 擴充。

### Heuristic Checks (啟發式檢查，僅回報)

這些任務高度依賴你的語意理解與邏輯判斷。請將發現的問題統整成清單並輸出給使用者，**絕對不可自動修復**：

- 各文章之間的事實矛盾或邏輯衝突。
- 已被較新原始資料（raw_sources）推翻的過時主張。
- 來源資料存在分歧，但文章內缺少 `Contradictions` 衝突註解的段落。
- 沒有來自其他 wiki 文章反向連結 (Backlinks) 的孤兒頁面。
- 引用的原始 wiki 來源發生大幅更新，導致其對應的 Archive (封存頁面) 結論可能已經過時。

### Post-Lint (檢查後)

附加本次檢查結果至 `wiki/log.md` 的最底部。確切格式請嚴格遵照 `references/log-template.md` 中的 Lint 區塊範例。

## Conventions (全域慣例與架構準則)

- **雙向連結優先 (Bi-directional Links)**：全面棄用傳統 Markdown 相對路徑。知識庫內部的所有實體參照（包含文章正文、YAML 陣列屬性、日誌紀錄），必須統一使用 Obsidian 風格的雙向連結語法 `[[檔名]]`。在對話輸出中回覆使用者時，也直接使用 `[[檔名]]` 即可。
- **扁平化目錄結構 (Flat Hierarchy)**：`wiki/` 與 `raw/` 預設僅支援「一層」主題子目錄（例如 `wiki/ai-agents/`）。絕對不要建立更深層的嵌套資料夾結構，請依賴 YAML 的 `tags` 屬性與知識圖譜來建立關聯。
- **時間戳記規範 (Timestamp Rules)**：
  - 日誌條目、`date_collected` (收集日) 和 `date_archived` (封存日) 必須使用當下日期 (YYYY-MM-DD)。
  - `date_updated` (更新日) 僅反映實體文章「知識內容」最後發生實質變更的時間。若僅是格式微調，不更新此欄位。
  - `date_published` (發布日) 來自原始資料，無法取得時填寫 `Unknown`。
- **元資料與狀態生命週期 (Metadata & Status Lifecycle)**：所有 Markdown 檔案必須具備 YAML Frontmatter 且標註 `status`，以決定你的讀寫權限：
  - `status: "ingested"` (raw 區)：具備絕對防寫保護，僅供讀取與萃取。
  - `status: "active"` (wiki 區)：核心知識文章，參與 Cascade Updates 連鎖更新。
  - `status: "archived"` (wiki 區)：特定時間點的封存快照，免疫連鎖更新。
  - `status: "stub"` (wiki 區)：由 Lint 建立的待補完空節點，等待未來知識注入。
- **檔案讀寫權限界線 (I/O Boundaries)**：
  - **Ingest**：必寫入實體文章與更新 `wiki/log.md`。嚴守目錄降載原則，僅在「建立新文章或新主題」時才更新 `wiki/index.md`。
  - **Query**：單純查詢時，預設僅在對話框輸出答案，絕對不寫入任何實體檔案。只有明確觸發 Archive (封存) 時，才會建立實體快照檔案，並同步更新 `index.md` 與 `log.md`。
  - **Lint**：必寫入 `wiki/log.md` 紀錄檢查結果。僅在執行決定性檢查時，才允許修改 `wiki/index.md` 與建立 stub 檔案。
