# split-book — 書籍前置切割器

## 角色分工

```
你（LLM）            腳本
─────────────        ──────────────────────────────────────
                  →  extract_headings.py  輸出標題清單
閱讀清單
[第一輪] 識別章邊界，建立確認表
[第二輪] 逐章決策：合併 / 保持 / 拆分
寫出 cuts.txt     →  split_book.py  機械式切割、寫出 md 檔
```

**腳本不呼叫任何 API；判斷邏輯完全由你負責。**

---

## 檔案路徑規範

| 路徑 | 說明 |
|------|------|
| `book/<書名>/<書名>.md` | 來源檔案（也接受 .txt） |
| `wiki/raw/<書名>_0.md` | 輸出（依序遞增） |
| `wiki/raw/<書名>_N_1.md` | 二次切割的子段 |
| `cuts.txt` | **暫存檔**，放根目錄，完成後必須刪除；禁止放入 book/ 或 wiki/ |

---

## 執行流程

### Step 1 — 萃取標題清單（只看前兩層）

第一輪分析只需要看「章 / 篇 / 部」這一層，加 `--level-max 2` 過濾掉圖表說明等雜訊：

```bash
python .gemini/skills/split-book/scripts/extract_headings.py <書名> --level-max 2
```

輸出範例：
```
# 書名：台灣AI大未來
# 總行數：2275
# 標題數量：42（level <= 2）
# 格式：行號<TAB>層級<TAB>標題

23	1	1:99的挑戰
107	2	PART I AI時代
109	2	第1章
111	2	電腦開始說人話，AI化身副駕
...
```

若標題數量仍然很多（> 80），可再縮小到 `--level-max 1`。
若書籍完全沒有 Markdown 標題（全文 level = 4+），改用預設值（不帶 --level-max）再觀察。

---

### Step 2 — 第一輪：識別章邊界，建立確認表

**目標：只做識別，不做決策。**

閱讀標題清單，找出書中頂層的章節分界。各書用語不同：
- 有的用「第N章」
- 有的用「第N部」、「第N篇」
- 有的用 PART I / PART II

識別原則：
- 選擇書中**最粗粒度的結構單元**作為切割基準
- 第一章之前的所有內容（書名、序言、推薦序、目錄）歸為前言區
- 每個章/篇/部對應一個切割點

識別完成後，**輸出確認表**（必要步驟，不可省略）：

```
| 序號   | 起始行 | 結束行 | 行數 | 章節名稱              |
|--------|--------|--------|------|-----------------------|
| 前言   | 0      | 22     | 23   | 書名/目錄/推薦序      |
| 第1章  | 23     | 106    | 84   | 1:99的挑戰            |
| 第2章  | 107    | 302    | 196  | PART I / 第1-2章      |
| 第3章  | 303    | ...    | ...  | ...                   |
```

確認表的「結束行」= 下一章的起始行（最後一章結束行 = 總行數）。
行數 = 結束行 - 起始行。

> 若標題層級混亂（如書中第3章以 level 4 標記），可用以下方式查看局部原文輔助判斷：
> ```bash
> # 查看第 300~320 行（PowerShell，行號從 1 開始）
> Get-Content book/<書名>/<書名>.md | Select-Object -Skip 300 -First 20
> ```

---

### Step 3 — 第二輪：逐章決策

依確認表的「行數」欄，逐行填入「決策」欄：

| 行數範圍 | 決策規則 |
|----------|----------|
| < 100 行 | **合併**：優先往後合併；若後一章合併後超過 600 行，改往前合併 |
| 100 ~ 600 行 | **保持** |
| > 600 行 | **拆分**：標記 `_SUB:N`，進行二次切割（見 Step 4b） |

在確認表加上「決策」欄後輸出：

```
| 序號   | 起始行 | 結束行 | 行數 | 章節名稱         | 決策          |
|--------|--------|--------|------|-----------------|---------------|
| 前言   | 0      | 22     | 23   | 書名/目錄        | 合併→第1章    |
| 第1章  | 23     | 106    | 84   | 1:99的挑戰       | 合併→第2章    |
| 第2章  | 107    | 302    | 196  | PART I / 第1-2章 | 保持          |
| ...    | ...    | ...    | ...  | ...             | ...           |
```

**合併後重新計算行數，確認合併後的區塊仍在 100~600 行之間。**

---

### Step 4a — 寫出 cuts.txt 並執行切割

決策確認後，依最終切割點建立 `cuts.txt`（格式：`行號<TAB>標題`，0-indexed）：

```
23	導言與前言
107	PART I AI時代 第1-2章
303	第3-4章 AI療癒/AI代理
...
```

先 dry-run 確認計畫：
```bash
python .gemini/skills/split-book/scripts/split_book.py <書名> cuts.txt --dry-run
```

dry-run 輸出中確認：
- 每個區塊行數都在合理範圍（沒有 < 50 行的孤立小塊）
- 章節標題對應正確

確認無誤後正式執行：
```bash
python .gemini/skills/split-book/scripts/split_book.py <書名> cuts.txt
```

輸出：
```
wiki/raw/<書名>_0.md   ← 前言（若第一個切割點不在第 0 行）
wiki/raw/<書名>_1.md   ← 第一個切割點
wiki/raw/<書名>_2.md   ← 第二個切割點
...
```

切割完成後，**立即刪除** cuts.txt：
```bash
Remove-Item cuts.txt
```

---

### Step 4b — 超大區塊的二次切割（視需要）

若 Step 3 有章節超過 600 行，對該章節查看 level 2 標題：

```bash
# 只查看該章節範圍的 level 2 標題
python .gemini/skills/split-book/scripts/extract_headings.py <書名> --level-max 2
# 搭配確認表中的起始/結束行，手動過濾出該章的標題行
```

依 level 2 標題決定二次切割點，在 cuts.txt 中用 `_SUB:N` 標記
（N 為該章節在主切割中的輸出編號）：

```
107	第一章
290	第二章
450	第二章 第二段	_SUB:2
560	第二章 第三段	_SUB:2
```

⚠️ 行號一律使用**原始檔案的行號**（0-indexed），不是子文本的行號。

重新執行（加 --force）：
```bash
python .gemini/skills/split-book/scripts/split_book.py <書名> cuts.txt --force
```

輸出命名結果：
```
wiki/raw/<書名>_2_0.md   ← 行 290~449
wiki/raw/<書名>_2_1.md   ← 行 450~559
wiki/raw/<書名>_2_2.md   ← 行 560~...
```

## 常見排查

| 症狀 | 處理方式 |
|------|----------|
| 未偵測到任何標題 | 改用預設 `--level-max 6`（不帶參數）重跑，或請使用者確認 md 格式 |
| 標題層級混亂 | 用 `Get-Content ... | Select-Object -Skip M -First N` 查看局部原文 |
| 已存在切割檔 | 詢問使用者是否加 `--force` 覆寫 |
| 某章節只有標題沒有內文 | 正常現象，切出的檔案行數很少，不影響後續流程 |
| dry-run 出現 < 50 行的區塊 | 回到確認表，將該區塊合併到相鄰章節 |
